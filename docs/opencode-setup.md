# OpenCode Setup

Status: supported.

OpenCode uses command-first activation. Agentic Loop generates repo-local
OpenCode agents plus a repo-local `/agenticloop` command. Normal OpenCode
prompts stay normal until you invoke that command.

Agentic Loop's own adapter settings stay in strict JSON (`agenticloop.json`
and `agenticloop/config.json`). Agentic Loop does not create, merge, validate, or
remove `opencode.jsonc`.

## Requirements

- Git repository
- GitHub CLI authenticated if using the GitHub backend
- OpenCode installed and connected to providers
- Agentic Loop files present in the target project

## Generate the Adapter

From the target project root:

```text
npx agenticloop generate opencode
```

This writes or refreshes exactly these Agentic Loop-owned files:

| Path | Purpose |
|---|---|
| `.opencode/agents/orchestrator.md` | Primary coordinator agent with OpenCode-native frontmatter. |
| `.opencode/agents/maintainer.md` | Maintainer subagent with OpenCode-native frontmatter. |
| `.opencode/agents/engineer.md` | Engineer subagent with OpenCode-native frontmatter. |
| `.opencode/commands/agenticloop.md` | Repo-local `/agenticloop` activation command for OpenCode. |

Each generated file contains a `<!-- Generated by Agentic Loop ... -->` banner
after the frontmatter. The banner records the source config key and the
refresh command. Validation warns if the banner is missing.

If you want scratch output, use `--output-dir <dir>`.

OpenCode may still have its own user-owned `opencode.jsonc`, but Agentic Loop
ignores it completely.

## Start the Loop

From the target project root in OpenCode, run:

```text
/agenticloop [task-id or task description]
```

The command invokes the generated `orchestrator` agent. It does not hard-code a
model. Model and reasoning settings still live in
`adapters.opencode.roleSettings.<role>` in `agenticloop.json`, which render to
the generated agent frontmatter `model` and `variant` fields.

During `agenticloop setup` or `agenticloop configure models --adapter opencode`,
the interactive picker first tries `opencode models` and uses those account-local
model IDs when available. If OpenCode is not installed, not connected, or returns
no parseable models, setup falls back to the bundled starter catalog and custom
model entry.

`agenticloop doctor` detects stale generated artifacts: if the model in a
generated `.opencode/agents/*.md` file disagrees with
`adapters.opencode.roleSettings` in `agenticloop.json`, the doctor output
reports the stale roles and suggests `agenticloop update --adapter opencode`.

The command body tells OpenCode to:

1. read `.agenticloop/project.md` first and stop for setup confirmation when `setup_status` is `unconfirmed`
2. follow `agenticloop/AGENTIC_LOOP.md` plus the canonical role contracts in `agenticloop/agents/`
3. create or refine the durable task record before implementation
4. route task-record, review, acceptance, and closeout work through maintainer
5. route scoped implementation and revision work through engineer

## Roles

OpenCode command-first output uses fixed agent names:

| Canonical role | OpenCode agent |
|---|---|
| `orchestrator` | `orchestrator` |
| `maintainer` | `maintainer` |
| `engineer` | `engineer` |

The generated frontmatter uses:

- `orchestrator`: `mode: primary`, `permission.edit: deny`, and
  `permission.task` allow rules for `maintainer` and `engineer`
- `maintainer`: `mode: subagent`
- `engineer`: `mode: subagent`

## Agentic Loop Procedures in Role Prompts

OpenCode is command-first and does not render a public Agentic Loop skill
surface. Instead, each generated role agent prompt points at the canonical
procedure files by explicit path, so the agent reads and follows the matching
file when a trigger applies:

```text
Agentic Loop internal procedures: when a trigger applies, read and follow the corresponding canonical file before acting:
- role-delegation: agenticloop/skills/role-delegation/SKILL.md
- blocked-state: agenticloop/skills/blocked-state/SKILL.md
```

The listed procedures come from each role's `requiredSkills`, and the path uses
configured `skills.sourceDirectory` (default `agenticloop/skills`). Any inline
`[[skill-name]]` markers in the canonical role body are also rewritten to the
same `agenticloop/skills/<skill-name>/SKILL.md` file-path references. This keeps
OpenCode pointed at the same canonical `agenticloop/skills/<name>/SKILL.md`
source the other hosts render, without copying per-procedure skills into a host
directory.

## Running OpenCode Against a Separate Target Project

When using Agentic Loop as an overlay in an existing target project, OpenCode
must be configured for and run from the target project root, not from the
Agentic Loop toolkit root.

The target repo needs its own generated `.opencode/agents/*.md` files and
`.opencode/commands/agenticloop.md`. Generate or refresh them with
`agenticloop generate opencode`.

## Role Delegation in OpenCode

The orchestrator is the primary agent. Maintainer and engineer are intended to
execute as separate subagent sessions, not as inline continuations of an
orchestrator message. After `/agenticloop` starts the task, delegated
maintainer and engineer sessions keep their own configured model and variant.

Agentic Loop treats any host mechanism that starts a separate role, task, or
subagent execution and accepts a role, agent, type, mode, or `subagent_type`
argument as real delegation. In OpenCode, the normal examples are Task-tool
subagent invocation and explicit `@maintainer` or `@engineer` invocation.

### Expected delegation behavior

1. The orchestrator identifies the task and decides which role to invoke.
2. It checks whether OpenCode exposes a real delegation path for that role.
   If the Task tool or explicit agent invocation is available, inline fallback
   is not allowed.
3. It uses OpenCode's actual delegation mechanism -- Task-tool subagent
   invocation or explicit `@maintainer` / `@engineer` invocation -- to create a
   visible subagent session.
4. The subagent session runs under the named agent's model, mode, and prompt.
5. The orchestrator waits for the subagent output before routing the next step.

OpenCode may support multiple visible subagent sessions, but Agentic Loop is
serial by default. For an authorized multi-task unit with 2 or more ready task
records, the orchestrator performs a Parallel Opportunity Scan before defaulting
to serial; bounded eligible batches may use up to 3 implementation lanes, and
choosing serial after eligible candidates exist requires a recorded concrete
reason. The orchestrator should not start parallel maintainer or
engineer sessions unless it has recorded the concurrency plan, collision
criteria, lease, and join condition required by `agenticloop/AGENTIC_LOOP.md`
and `agenticloop/skills/role-delegation/SKILL.md`. Long-running parallelism has
stronger observability requirements than short bounded join-based batches.
Parallel write lanes that
mutate repository files require a separate `git worktree` and branch per lane;
a branch alone is not sufficient in a shared checkout. The lease progress
checkpoint is a return-to-orchestrator checkpoint cadence, not an async
heartbeat, unless the host exposes running-subagent status.

Launch OpenCode sessions that run Agentic Loop with non-interactive Git
environment variables (`GIT_EDITOR=true`, `GIT_SEQUENCE_EDITOR=true`,
`GIT_PAGER=cat`, `GIT_TERMINAL_PROMPT=0`, `GH_EDITOR=true`, `GH_PAGER=cat`,
`GH_PROMPT_DISABLED=1`) as described in
[Host Adapters](host-adapters.md#non-interactive-git-environment). This prevents
unattended lanes from blocking on `COMMIT_EDITMSG`, interactive rebase todo
files, pagers, PR prompts, or credential prompts.

Prose in the orchestrator output describing what maintainer or engineer will do
is not delegation.

## Project Skills

OpenCode may expose target-project skills that are visible in the host. Those
project skills may be used for domain-specific procedures when their triggers
apply. Agentic Loop skills remain authoritative for task records, evidence,
review, blocked state, and closeout.

If Codex output is also generated in the same target, OpenCode can see
`.agents/skills/agenticloop/SKILL.md`. That is not fatal, but Agentic Loop's
supported OpenCode activation remains `/agenticloop`.
